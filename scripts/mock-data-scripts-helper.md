# Помощник по mock-скриптам

Этот файл описывает два вспомогательных скрипта для генерации mock-данных, которые можно читать с сервера как «имитацию БД».

## 1. Генератор событий журнала

Скрипт создает файл с журналом событий за выбранный период.

### Назначение
Подходит для:
- тестирования `/api/v2/journal`
- проверки пагинации по `before` курсору
- проверки фильтрации по `severity` и `category`
- отладки таблицы журнала и drawer с деталями события

### Предполагаемая модель события
События генерируются через функцию вида:

```js
generateJournalMessage(ts)
```

Где:
- `ts` — timestamp события
- `id` внутри события должен строиться из этого же времени, например через `uuidv7(...)`
- итоговый объект содержит как минимум:
  - `id`
  - `ts`
  - `category`
  - `event`
  - `severity`
  - `message`
  - `ack`
  - `actor`
  - `payload`

### Рекомендуемый формат файла
Обычно удобно хранить так:

```json
{
  "schemaVersion": 1,
  "generatedAt": 1710000000000,
  "meta": { ... },
  "items": [ ... ]
}
```

Либо в формате `jsonl`, если нужен построчный разбор.

### Важная идея
Сначала генерируется `ts`, потом на его основе создается событие:

```js
const ts = randomTimestampBetween(fromMs, toMs);
const item = generateJournalMessage(ts);
```

Это нужно, чтобы:
- `ts` был согласован с `id`
- курсоры работали стабильно
- сортировка по времени и `uuidv7` не расходилась

### Что потом делает сервер
Сервер:
- читает файл
- фильтрует по диапазону времени
- фильтрует по `severity/category`
- сортирует по `ts desc`
- применяет курсор `before`
- возвращает страницу событий

## 2. Генератор данных для графиков

Скрипт создает файл с временными рядами для нескольких переменных вида:

- `variable1`
- `variable2`
- `variable3`
- ...

### Назначение
Подходит для:
- тестирования `/api/v2/graph`
- наполнения графиков данными, похожими на реальные
- отладки Chart.js, legend, time scale, period mode и real-time mode

### Формат записи
Каждая запись представляет одну точку измерения:

```json
{
  "ts": 1710000000000,
  "variable": "variable1",
  "value": 52.314
}
```

### Почему именно такой формат
Этот формат удобен, потому что сервер потом может:
- фильтровать по `from/to`
- фильтровать по списку переменных
- ограничивать число точек
- агрегировать или прореживать данные
- собирать готовые `datasets` для Chart.js

### Типы сигналов
Скрипт случайно выбирает профиль сигнала для каждой переменной. Например:
- `sine`
- `noisySine`
- `step`
- `trend`
- `binary`
- `spike`

Это делает графики более живыми и похожими на реальные сигналы.

### Что записывается дополнительно
В `meta` можно сохранять описание переменных:

```json
{
  "variable": "variable1",
  "kind": "step",
  "params": { ... }
}
```

Это полезно для отладки, чтобы понимать, почему конкретная переменная выглядит именно так.

### Что потом делает сервер
Сервер:
- читает файл
- выбирает точки по диапазону времени
- отбирает только нужные переменные
- при необходимости ограничивает число точек
- преобразует точки в структуру для Chart.js:

```js
{
  label: "variable1",
  data: [
    { x: 1710000000000, y: 10.5 },
    { x: 1710000060000, y: 10.9 }
  ]
}
```

## Практические рекомендации

### Для журнала
Лучше хранить:
- `ts` как число в миллисекундах UTC
- `id` как `uuidv7`, связанный с `ts`

На фронте:
- не подменять `ts`
- форматировать локальное время отдельно, например в `tsText`

### Для графиков
Лучше хранить:
- `ts`
- `variable`
- `value`

А уже на сервере или фронте собирать данные под нужный формат графика.

### Когда использовать `json`
Используй `json`, если:
- хочешь просто быстро прочитать весь файл
- объем данных не очень большой
- удобно хранить `meta` рядом с `items`

### Когда использовать `jsonl`
Используй `jsonl`, если:
- данных много
- нужен построчный разбор
- в будущем хочешь читать файл потоково

## Минимальный workflow

### Журнал
1. Сгенерировать файл событий
2. Подключить серверный endpoint чтения файла
3. На фронте отображать `ts` через локальный formatter

### Графики
1. Сгенерировать файл временных рядов
2. Подключить endpoint чтения точек по `from/to/variables`
3. Преобразовать ответ в `datasets` для Chart.js

---

## Идея на будущее
Оба файла можно воспринимать как временный file-based repository.

Позже источник данных можно заменить:
- на SQLite
- на Timescale / PostgreSQL
- на отдельный сервис хранения

При этом HTTP-контракт для фронта можно почти не менять.

## Подсказки по использованию

Ниже — краткая памятка: как запускать скрипты и какие аргументы они принимают.

### Общий вид запуска

Оба скрипта запускаются через Node.js:

```bash
node <путь-к-скрипту> [аргументы]
```

Например:

```bash
node scripts/generate-journal-db.js --days 7 --out ./data/mock-journal.json
```

```bash
node scripts/generate-timeseries-db.js --variables 5 --points 300 --out ./data/mock-timeseries.json
```

---

## Генератор событий журнала

### Пример вызова

```bash
node scripts/generate-journal-db.js --count 1000 --days 7 --out ./data/mock-journal-db.json
```

### Пример с явным периодом

```bash
node scripts/generate-journal-db.js \
  --count 3000 \
  --from 2026-03-01T00:00:00Z \
  --to 2026-03-19T23:59:59Z \
  --out ./data/mock-journal-db.json
```

### Основные аргументы

- `--out`  
  Путь к выходному файлу.  
  Пример: `--out ./data/mock-journal-db.json`

- `--count`  
  Сколько событий сгенерировать.  
  Пример: `--count 1000`

- `--days`  
  За сколько последних дней генерировать данные, если не указаны `--from` и `--to`.  
  Пример: `--days 7`

- `--from`  
  Начало периода в ISO 8601 UTC.  
  Пример: `--from 2026-03-01T00:00:00Z`

- `--to`  
  Конец периода в ISO 8601 UTC.  
  Пример: `--to 2026-03-19T23:59:59Z`

- `--format`  
  Формат файла: `json` или `jsonl`.  
  Пример: `--format jsonl`

- `--sort`  
  Порядок сортировки: `asc` или `desc`.  
  Пример: `--sort desc`

### Когда использовать `--days`, а когда `--from/--to`

- Если нужен быстрый мок «за последние N дней» — используй `--days`
- Если нужен точный период — используй `--from` и `--to`

## Генератор данных для графиков

### Пример вызова

```bash
node scripts/generate-timeseries-db.js --variables 5 --points 300 --days 7 --out ./data/mock-timeseries-db.json
```

### Пример с JSONL

```bash
node scripts/generate-timeseries-db.js \
  --variables 8 \
  --points 1000 \
  --days 30 \
  --format jsonl \
  --out ./data/mock-timeseries-db.jsonl
```

### Пример с явным периодом

```bash
node scripts/generate-timeseries-db.js \
  --variables 6 \
  --points 500 \
  --from 2026-03-01T00:00:00Z \
  --to 2026-03-19T23:59:59Z \
  --out ./data/mock-timeseries-db.json
```

### Основные аргументы

- `--out`  
  Путь к выходному файлу.  
  Пример: `--out ./data/mock-timeseries-db.json`

- `--variables`  
  Количество переменных. Будут созданы имена вида `variable1`, `variable2`, ...  
  Пример: `--variables 5`

- `--points`  
  Количество точек на каждую переменную.  
  Пример: `--points 300`

- `--days`  
  За сколько последних дней генерировать данные, если не указаны `--from` и `--to`.  
  Пример: `--days 7`

- `--from`  
  Начало периода в ISO 8601 UTC.  
  Пример: `--from 2026-03-01T00:00:00Z`

- `--to`  
  Конец периода в ISO 8601 UTC.  
  Пример: `--to 2026-03-19T23:59:59Z`

- `--format`  
  Формат файла: `json` или `jsonl`.  
  Пример: `--format json`

- `--jitter`  
  Небольшой случайный сдвиг точки по времени относительно идеального шага.  
  Полезно, чтобы данные выглядели менее искусственно.  
  Пример: `--jitter 0.05`

### Как выбирать параметры

- Для легкого теста UI:  
  `--variables 3 --points 100`

- Для более реалистичного графика:  
  `--variables 5 --points 300`

- Для тяжелого теста таблиц, виртуализации, downsampling:  
  `--variables 10 --points 5000`

## Полезные замечания

- Если `--from` и `--to` не указаны, используется период от `now - days` до `now`
- Для больших файлов удобнее `jsonl`
- Для простого локального чтения и наличия `meta` удобнее `json`
- Если путь к папке не существует, скрипт должен создать ее автоматически
- Лучше использовать UTC в аргументах `--from` и `--to`

### Хороший пример ISO-строки

```text
2026-03-19T23:59:59Z
```

### Менее желательный вариант

```text
2026-03-19
```

Такой формат может интерпретироваться менее очевидно, поэтому для серверных моков лучше всегда писать полную UTC-строку.
