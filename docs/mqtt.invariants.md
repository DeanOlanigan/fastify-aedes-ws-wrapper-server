# Архитектурные инварианты MQTT-команд и квитирования журнала

## Контекст

В текущей реализации HTTP API и MQTT broker находятся в одном процессе. Это удобно для демо и переходного этапа, но архитектура должна уже сейчас строиться так, чтобы в будущем broker можно было вынести в отдельную службу без пересмотра базовой модели безопасности и контрактов.

Цель текущего этапа:

* укрепить контракт команд;
* разделить бизнес-авторизацию и broker-level authorization;
* оставить MQTT как транспорт событий и внутренних команд;
* перевести пользовательские команды на доверенный серверный путь.

---

## Инварианты доверия

### 1. Пользователь не является доверенным MQTT-издателем команд

Браузерный клиент не должен считаться доверенным источником командных сообщений.

Следствия:

* UI может подписываться на realtime-события;
* UI не должен напрямую публиковать доменные команды в `commands/#`;
* любое ограничение `publish` только на уровне frontend-кода не является защитой.

### 2. Единственная доверенная точка входа пользовательской команды — HTTP API

Пользовательское действие, требующее прав доступа, должно сначала пройти через HTTP API.

Следствия:

* route проверяет сессию, роль и право;
* route валидирует входные данные;
* route формирует доверенную внутреннюю команду;
* только после этого команда попадает в MQTT.

### 3. Бизнес-авторизация и broker-авторизация — разные уровни

Нельзя смешивать вопрос «может ли пользователь выполнить бизнес-операцию» с вопросом «может ли MQTT-клиент писать/читать определенный topic».

HTTP/API отвечает за:

* аутентификацию пользователя;
* RBAC/rights;
* step-up, если понадобится;
* аудит и доменную валидацию.

Broker отвечает за:

* аутентификацию MQTT-клиента;
* topic-level ACL;
* разделение технических клиентов по ролям.

### 4. `requestedBy` в команде — это audit context, а не доказательство прав

Поле `requestedBy` описывает, от чьего имени была инициирована команда, но не является источником истины для проверки прав.

Следствия:

* права уже должны быть проверены до публикации команды;
* consumer команды доверяет `requestedBy` только потому, что команда пришла от доверенного API/сервиса;
* если недоверенный клиент сможет публиковать такую команду, `requestedBy` становится подделываемым.

### 5. `publishInternal()` — текущая форма доверенной публикации

Пока broker и API находятся в одном процессе, серверная trusted-публикация выполняется через `publishInternal()`.

Следствия:

* это архитектурный seam для будущей миграции;
* в будущем `publishInternal()` может быть заменен на публикацию в удаленный broker от имени сервисного MQTT-клиента;
* HTTP routes не должны зависеть от сырого broker API сильнее, чем это необходимо.

## Инварианты по командам

### 6. У всех команд единый верхнеуровневый формат

Команда должна иметь единый конверт:

```json
{
  "commandId": "uuid-v7",
  "type": "journal.ack",
  "requestedAt": 1730000000000,
  "requestedBy": {
    "userId": "u1",
    "login": "operator",
    "name": "Иван Петров"
  },
  "payload": {
    "eventId": "evt-1",
    "event": "security.roles.changed",
    "message": "Роли пользователей обновлены"
  }
}
```

Верхний уровень отвечает за:

* корреляцию (`commandId`);
* тип команды (`type`);
* момент запроса (`requestedAt`);
* контекст инициатора (`requestedBy`).

`payload` отвечает за доменную нагрузку конкретной команды.

### 7. `payload` — это снимок цели команды, а не абсолютная истина

Поля внутри `payload` допустимо использовать как контекст команды, но не как единственный источник истины в долгосрочной архитектуре.

Для `journal.ack`:

* `payload.eventId` — главный идентификатор цели;
* `payload.event` — снимок кода события на момент отправки;
* `payload.message` — снимок сообщения на момент отправки.

В будущем, если будет store/DB журнала, обработчик может сверять эти данные с записью по `eventId`.

### 8. Минимальный обязательный контракт `journal.ack`

Обязательные поля:

* `commandId`
* `type === "journal.ack"`
* `requestedAt`
* `payload.eventId`

Желательные, но допускающие `null`:

* `requestedBy`
* `payload.event`
* `payload.message`

### 9. Команда должна быть валидируема независимо от transport-слоя

Consumer команды обязан проверять хотя бы минимальную форму команды:

* корректный `type`;
* наличие `commandId`;
* наличие `payload.eventId`;
* корректный JSON payload.

Даже доверенная команда должна проходить контрактную валидацию.

## Инварианты по journal-событию после квитирования

### 10. Результирующее journal-событие должно соответствовать общей схеме журнала

Событие квитирования публикуется как обычная запись журнала, а не как специальный технический формат.

Пример:

```json
{
  "$schema": "./journal.event.schema2.json",
  "schemaVersion": 1,
  "id": "uuid-v7",
  "ts": 1730000000000,
  "severity": "info",
  "category": "event",
  "event": "event.acknowledged",
  "actor": {
    "type": "user",
    "id": "u1",
    "login": "operator",
    "name": "Иван Петров"
  },
  "ack": null,
  "payload": {
    "targetEvent": {
      "id": "evt-1",
      "event": "security.roles.changed",
      "message": "Роли пользователей обновлены"
    }
  },
  "message": "Иван Петров квитировал событие security.roles.changed: Роли пользователей обновлены"
}
```

### 11. `actor` в journal-событии строится из `requestedBy`

Если команда прошла через доверенный HTTP/API путь, то `requestedBy` становится источником для поля `actor`.

Сопоставление:

* `requestedBy.userId` → `actor.id`
* `requestedBy.login` → `actor.login`
* `requestedBy.name` → `actor.name`
* `actor.type` для текущего сценария фиксируется как `user`

### 12. `payload.targetEvent` содержит описание цели квитирования

Результирующее journal-событие должно содержать внутри `payload.targetEvent`:

* `id`
* `event`
* `message`

Даже если пока используется демо-режим и данные приходят из самой команды, формат должен оставаться стабильным.

### 13. Человекочитаемое поле `message` формируется сервисом централизованно

Итоговый текст journal-события должен строиться на сервере, а не приходить готовым из клиента.

Причины:

* единый стиль сообщений;
* защита от недостоверного текста с клиента;
* возможность локально переопределять шаблон сообщений.

## Инварианты по слоям ответственности

### 14. HTTP route не исполняет команду, а только принимает и публикует ее

`journalRoutes` не должен сам выполнять квитирование.

Route отвечает за:

* валидацию HTTP body;
* проверку прав;
* формирование команды;
* trusted-публикацию в MQTT.

### 15. Dispatcher не реализует доменную логику

`createCommandDispatcher()` не должен знать детали журнала, БД и текстов сообщений.

Он отвечает за:

* распознавание типа команды;
* первичную контрактную валидацию;
* вызов соответствующего command service.

### 16. Domain service исполняет команду и порождает результат

`journalCommands.ackEvent()` — это место, где исполняется доменная команда.

Сервис отвечает за:

* интерпретацию payload;
* подготовку результирующего journal-события;
* публикацию результата;
* в будущем — проверку store/DB, идемпотентность и фиксацию ack.

## Инварианты безопасности

### 17. Запрет `publish` в UI — это удобство, а не защита

Даже если клиентский код не вызывает `publish`, внешний MQTT-клиент все равно сможет попытаться подключиться к broker и отправить команду.

Следствие:

* реальная защита должна быть на стороне broker ACL и/или отсутствия прямого пользовательского доступа к command topics.

### 18. До разделения broker/API допускается серверная trusted-публикация без broker ACL для внутреннего пути

Пока broker и API находятся в одном процессе, допустимо использовать локальную trusted-публикацию для внутренних команд.

Но это не отменяет будущей необходимости:

* аутентификации MQTT-клиентов;
* topic ACL;
* разделения сервисных и UI-клиентов.

### 19. После выноса broker в отдельную службу API становится сервисным MQTT-клиентом

При переходе на отдельный broker модель должна остаться той же:

* пользователь по-прежнему ходит в HTTP API;
* API публикует команды в broker как доверенный сервис;
* broker проверяет права сервисного клиента на topics.

### 20. Consumer команды не обязан заново вычислять пользовательские RBAC-права

Command worker может и должен проверять:

* валидность контракта;
* существование целевого объекта;
* допустимость текущего состояния;
* идемпотентность.

Но пользовательская авторизация остается ответственностью HTTP/API слоя.

## Инварианты эволюции архитектуры

### 21. Текущий контракт должен пережить переход от embedded broker к отдельному broker

Смена транспорта trusted-публикации не должна требовать менять JSON-контракт команды и результатирующего journal-события.

### 22. Абстракция trusted command publishing должна сохраниться

Сейчас это `mqttPublishInternal()`.
В будущем это может стать `commandBus.publish(...)` или MQTT client API-сервиса.

Главное: routes не должны быть привязаны к конкретной локальной реализации broker.

### 23. Топики команд и событий должны оставаться логически разделенными

Даже если пока итоговое journal-событие публикуется в `journal`, логическое разделение уже должно подразумеваться:

* команды — это `commands/...`
* события/записи — это `journal`, `events/...` или другой read-only namespace

### 24. Контракт важнее текущей демо-реализации

Даже если часть логики пока работает как stub:

* формат команды должен быть стабильным;
* формат journal-события должен быть стабильным;
* слои ответственности не должны смешиваться только ради демо-удобства.

---

## Практические следствия для текущего кода

1. `POST /api/v2/journal/ack` остается единственным пользовательским входом для квитирования.
2. Route обязан использовать `requireAuth` и `requireRight("journal.ack")`.
3. Route публикует команду формата `journal.ack` через `fastify.mqttPublishInternal(...)`.
4. `createCommandDispatcher()` валидирует контракт и передает управление `journalCommands.ackEvent(...)`.
5. `journalCommands.ackEvent(...)` публикует запись журнала по общей схеме.
6. UI должен получать результат как journal-событие, а не как прямой ответ от MQTT-команды.
7. В будущем `mqttPublishInternal()` может быть заменен без изменения route-контракта.

* Как выглядит источник истины для journal event по `eventId` — store, DB или stream cache.
* Как обрабатывать повторное квитирование: `noop`, `ignored`, `rejected` или отдельное journal-событие.
* Нужен ли отдельный result topic для command processing или достаточно итоговой journal-записи.
* Нужно ли хранить `ack`-метаданные внутри исходного события и в каком формате.
* Какой namespace топиков будет финальным в продакшен-реализации.
